# Release pipeline — prototype findings + blockers

Generated by the v1 release-pipeline prototype on 2026-05-15.

## What was prototyped

`scripts/release-prototype/verify-roundtrip.sh` — a local driver that
exercises the production updater verify path (cosign sign-blob → raw
.sig → cosign verify-blob → tampered-tarball rejection) against a
synthetic release built from the working tree. It also drives
`hal0.updater.Updater.check()` end-to-end against a `file://` manifest.

Result: **PASSED** locally on hal0-dev. cosign 3.0.6, project venv.

`.github/workflows/release.yml` — draft workflow that, on a `v*` tag
push, builds the tarball, cosign-signs it keyless via GH Actions OIDC,
generates the `hal0.releases.v1` manifest, and uploads all three to a
GitHub Release. **Not yet pushed; not yet triggered.**

## What's verified

- The legacy `cosign verify-blob --signature <file>` invocation (which
  `src/hal0/updater/updater.py:_verify_cosign` uses) still works in
  cosign 3.x.
- `ReleaseManifest` (pydantic) accepts the manifest shape
  `release.yml` will publish.
- Tampered tarballs are rejected at both layers — sha256 in the updater,
  signature in cosign.
- The artifact layout (`hal0-<ver>/` top-level dir) matches what
  `_extract_tarball` expects.

## Findings

### 1. cosign 3.x changed defaults

`cosign sign-blob` defaults to `--new-bundle-format=true` and silently
ignores `--output-signature`. To keep emitting the legacy raw `.sig`
file the updater currently consumes, the workflow passes
`--new-bundle-format=false`. Documented inline in `release.yml`.

**Follow-up (not a v1 blocker):** teach `_verify_cosign` to also accept
a Sigstore Bundle (`*.bundle` JSON with `messageSignature` +
`verificationMaterial`). That's the future-proof path; the legacy raw
`.sig` works for now.

### 2. The `signer_identity` regex must match exactly twice

The workflow writes the regex into the manifest and, in the same step,
self-verifies with that same regex. The two strings must stay in
lock-step or the very first real release won't pass its own
verification. They are anchored to `${GITHUB_REF}` (i.e.
`refs/tags/vX.Y.Z`) so each release pins to its own workflow ref.

### 3. `manifest.json` toolbox digests must be pinned BEFORE tagging

`release.yml` refuses to publish if any `toolbox_images.<n>.digest` is
null. That means `toolbox.yml` must have run on `main` and committed
the patched `manifest.json` first. `flm` is currently null
(stretch) — that will block a real `v1.0.0-rc1` cut unless we either
ship without flm or stub a placeholder digest.

## Blockers for a real v1.0.0-rc1 cut

### Hard blockers

1. **`releases.hal0.dev` does not exist.** The updater hits
   `https://releases.hal0.dev/{channel}.json` by default. Until that
   host serves the manifest, users have to set `HAL0_RELEASES_URL` to
   the GitHub-Release-asset URL. Decision needed: stand up
   `releases.hal0.dev` (Cloudflare Pages? S3 + CloudFront? GitHub
   Pages on a custom domain?) **before** v1, or ship v1 with the
   `HAL0_RELEASES_URL` workaround documented in the README.

2. **`flm` toolbox image has no digest** (`manifest.json:18`). Either:
   - drop FLM from the v1 toolbox set and re-add post-launch (PLAN
     already calls FLM a stretch), or
   - publish a real FLM image so the digest gets pinned by
     `toolbox.yml`.

3. **`HAL0_UPDATE_SKIP_COSIGN=1` escape hatch exists.** Documented gap
   in `docs/internal/release-manifest.md` — must close before v1 ships. Decision:
   keep for `v1.0.0-rc*` (so smoke testing works on the LXC without a
   real release) and remove in `v1.0.0` proper, or remove now and
   require a real release for any LXC smoke.

### Soft blockers / decisions

4. **Tag-cutting ritual.** `scripts/release-check.sh` already exists and
   gates pyproject version, manifest pinning, lint, tests, and
   release-test report freshness. The doc trail should say "always
   `bash scripts/release-check.sh --tag vX.Y.Z` before
   `git tag vX.Y.Z`". Worth adding to CONTRIBUTING.md.

5. **`release.yml` self-installs the package** (`pip install -e
   ".[dev]"`) just to validate the manifest against the runtime schema.
   Adds ~30s. Trade-off: we catch shape regressions before publish.
   Acceptable.

6. **Concurrency `cancel-in-progress: false`** — never cancel a
   half-published release. Confirm this is the right call (worst case:
   two workflow runs racing on the same tag will both try to upload
   assets, and `--clobber` wins; either way, no inconsistent state).

## To exercise the verify path against a real release

Once the first `release.yml` run completes:

```bash
# Point the updater at the GH Release URL until releases.hal0.dev exists.
HAL0_RELEASES_URL="https://github.com/hal0ai/hal0/releases/download/v0.1.0-rc1/stable.json" \
    hal0 update --check

# Or, full apply (will atomic-swap /usr/lib/hal0/current — root needed):
sudo HAL0_RELEASES_URL="…" hal0 update
```

If this works against a real GH-Actions-signed artifact, the cosign
keyless OIDC path is proven. That's the last unverified step.
