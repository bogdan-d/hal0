# ADR 0009 — FLM trio: pack three NPU roles into one process, bypass Lemonade for asr/embed dispatch

- **Status:** Accepted
- **Date proposed:** 2026-05-22
- **Date accepted:** 2026-05-22
- **Implementing PRs:** PR-19 (FLM trio dispatch — capability layer routes `stt-npu`/`embed-npu` directly to FLM child port discovered from `/v1/health`), PR-20 (NPU exclusivity validation + "swap incoming" UX) per the lemonade-adoption-plan §11
- **Depends on:** ADR-0008 (Lemonade adoption)
- **Touches:** CONTEXT.md `FLM trio` entry, `slot type` entry, `default slot` entry

## Context

Three independent constraints, stacked, force a non-obvious architecture for hal0's NPU surface. Each was discovered or verified empirically during the 2026-05-22 spike on LXC 105 (Strix Halo NPU passthrough).

### Layer 1 — hardware: one AMDXDNA HW context per host

The Strix Halo XDNA2 NPU exposes exactly one hardware context per host. A second `flm serve` invoked while a first is already running fails with:

```
DRM_IOCTL_AMDXDNA_CREATE_HWCTX IOCTL failed err=-22
```

Verified empirically. There is no kernel-side multiplexing; this is the silicon's contract. Two NPU-serving processes is not a configuration we can choose — it is unreachable. Per-host singleton.

### Layer 2 — FLM upstream: one process, multiple roles via flags

FLM v0.9.42 documents two flags that pack additional model roles into a single `flm serve` invocation:

- `--asr 1` — load a transcription model alongside the chat model
- `--embed 1` — load an embedding model alongside the chat model

All three model roles share the NPU's 8 columns within the single AMDXDNA HW context. Verified 2026-05-22 (spike Phase C + the trio side-experiment immediately after): `flm serve gemma3:1b --asr 1 --embed 1` brings up gemma3:1b chat (40 tok/s, prefill 16 tok/s, KV occupancy 0.66) plus Whisper-V3-Turbo transcription plus Embedding-Gemma-300M (768-dim) — all loaded simultaneously, ~2 GB NPU memory total, no eviction thrash between them.

The trio is real. The NPU is not a single-model device in practice — it is a single-process device.

### Layer 3 — Lemonade: WrappedServer is a one-model abstraction

Lemonade's `backends/fastflowlm_server.cpp` wraps `flm serve` as a `WrappedServer`. Each WrappedServer carries one Lemonade-registered model name; Lemonade's `/v1/load` adds/replaces a WrappedServer; `/v1/health.loaded[]` reports one entry per WrappedServer; `/v1/unload` removes one.

Crucially:

- Lemonade does NOT pass `--asr 1` or `--embed 1` by default. The wrapper spawns `flm serve <model>` with no trio flags.
- The wrapper assumes one FLM model per process. There is no Lemonade API surface for "this WrappedServer has three roles."
- Lemonade enforces NPU exclusivity at the WrappedServer level: a second NPU-typed load evicts the first FLM WrappedServer before spawning a replacement.

So if we adopt Lemonade verbatim, the NPU serves at most one model role at a time — chat OR stt OR embed, never two, never three. We lose ~2/3 of what the hardware can actually do.

### Three architecture options considered

| Option | Shape | Verdict |
|---|---|---|
| **W — accept Lemonade's limit** | NPU serves one role at a time; hal0 picks chat by default; users wanting NPU stt give up NPU chat | Rejected. Leaves the hardware idle. Users who paid for a Strix Halo NPU expect to use it for the workloads it was built for. |
| **X — flm.args + hal0 fan-out** | Set Lemonade's `flm.args = "--asr 1 --embed 1"`; expose three slots; hal0 dispatches asr/embed directly to the FLM child port; chat goes through Lemonade normally | **Picked.** Lemonade still owns FLM lifecycle (one WrappedServer, one chat model, eviction policy). The extra two roles are coresident free-riders. |
| **Y — bypass Lemonade entirely for NPU** | hal0 manages a parallel `flm serve` process outside Lemonade; capability layer routes all NPU traffic directly | Rejected. Doubles the lifecycle surface (hal0 now owns FLM process management AND Lemonade owns the GPU stack); breaks ADR-0008's "Lemonade owns inference lifecycle" thesis far more than X does; loses Lemonade's `/v1/health` + `/logs/stream` integration for the FLM child. |

X is the minimum departure from ADR-0008 that captures the hardware's real ceiling.

## Decision

Adopt option X. Three pieces:

### 1. Set Lemonade's `flm.args` to enable the trio

In `/var/lib/hal0/lemonade/config.json`, generated by install.sh PR-5:

```json
{
  "flm": { "args": "--asr 1 --embed 1" }
}
```

This is plumbed into every `flm serve` Lemonade spawns. Lemonade still thinks it's running a chat model — it has no schema for the other two roles — but the FLM child process honours both flags and loads all three.

### 2. Three slots in `capabilities.toml`, all backing the same FLM child

Per CONTEXT.md `FLM trio` table:

| Slot | type | device | Backing |
|---|---|---|---|
| `agent` | `llm` | `npu` | the FLM trio's chat model (the one Lemonade knows about) |
| `stt-npu` | `transcription` | `npu` | the trio's `--asr` model (Lemonade does NOT know) |
| `embed-npu` | `embedding` | `npu` | the trio's `--embed` model (Lemonade does NOT know) |

Loading any one of the three starts the FLM trio process. The other two roles are "available" instantly — they were loaded at FLM startup, not on demand. There is no per-role load step.

### 3. Direct child-port dispatch for asr/embed; Lemonade routing for chat

hal0's capability dispatcher reads `/v1/health.loaded[].backend_url` for the FLM model and routes by request type:

```python
# src/hal0/capabilities/dispatch.py (sketch)
def dispatch(slot: Slot, request: Request) -> Response:
    if slot.device == "npu" and slot.type in ("transcription", "embedding"):
        flm_url = lemonade_client.flm_backend_url()  # from /v1/health.loaded[]
        path = {
            "transcription": "/v1/audio/transcriptions",
            "embedding":     "/v1/embeddings",
        }[slot.type]
        return http.post(f"{flm_url}{path}", json=request.body)

    # everything else (chat, GPU slots, etc.) goes through Lemonade normally
    return lemonade_client.proxy(slot, request)
```

Chat-to-NPU still goes through Lemonade's `/v1/chat/completions` — Lemonade routes that to the FLM child port itself, and the chat WrappedServer is exactly what Lemonade's auto-load + eviction logic expects.

The asr/embed paths are the carve-out. They bypass Lemonade because Lemonade does not know those endpoints exist on that child. From Lemonade's POV, the FLM child serves one chat model; from hal0's POV, the FLM child serves three OpenAI-compatible endpoints, and we hit the two Lemonade is unaware of directly.

## Consequences

### Positive

- **Full NPU multi-role utilisation.** Chat + transcription + embedding all live on the NPU at once. ~2 GB NPU memory budget for the trio leaves headroom for a Pro-tier user to keep the rest of the LRU on GPU. The hardware's actual ceiling is what we ship.
- **No eviction thrash for the NPU multi-modality use case.** Switching between "answer a question" and "transcribe this audio clip" and "embed this doc" never causes a model load — they are coresident at FLM startup.
- **Free upgrade path.** When Lemonade upstream ships native multi-model FLM support (the WrappedServer schema gains a roles field, or `flm` becomes three WrappedServers sharing one process), hal0's dispatcher drops the direct-port branch and falls back to Lemonade for everything. The slot surface in `capabilities.toml` does not change.
- **Slot identity stays clean.** Three real slots with three types and three default-per-type slots — same shape as GPU. Users do not see "the NPU is a special thing"; they see three NPU rows in the dashboard, each enableable, each with a model picker (or, for stt-npu/embed-npu, a "coresident with NPU chat" marker).
- **NPU stays a real product surface, not a footnote.** Pro+ bundle tiers can offer "Whisper-V3-Turbo on NPU + Embedding-Gemma on NPU + Qwen on GPU" without any "but the NPU is one-model-at-a-time" caveat.

### Negative / costs

- **NPU LLM swap is slow (~14s).** Changing the `agent` slot's chat model means tearing down the FLM trio (stt + embed go with it) and restarting `flm serve <new-chat-model> --asr 1 --embed 1`. UI must surface "swap incoming, voice + embed paused" — PR-20 ships that UX.
- **hal0 carries direct-dispatch logic.** A small but real chunk of the capability dispatcher exists solely to route around Lemonade for two endpoint paths. New developers will see it and ask "why?" — this ADR is the answer.
- **ADR-0008's "Lemonade owns inference lifecycle" claim has a documented carve-out.** Lemonade owns the FLM process lifecycle; hal0 owns the routing for two endpoints that share that process. The carve-out is narrow (no parallel FLM process, no NPU-side state outside Lemonade's `/v1/health` view), but it exists. Future ADRs touching lifecycle ownership must read this one.
- **Future FLM upstream changes to flag semantics could break the trio assumption.** If `--asr 1` becomes `--asr <path-to-model>` in v0.10, or if the flag is renamed, hal0's install.sh `flm.args` value silently produces a one-model FLM child instead of erroring. Mitigation: install-time `flm validate` plus a runtime probe that asserts `/v1/audio/transcriptions` returns non-501 on the FLM child after first load. Caught early, fixed by bumping the pinned FLM .deb version + `flm.args`.
- **Only `agent` is reflected in `/v1/health.loaded[]`.** Lemonade reports one entry for the FLM model. hal0's UI must synthesise `stt-npu` and `embed-npu` "loaded" state from the agent slot's state plus a runtime probe of the two endpoints. Dashboard adds a "coresident" badge so users understand why two slots flip green/red together.

### Risks

- **NPU exclusivity validation must be strict.** Two `device = "npu", type = "llm"` slots `enabled = true` simultaneously is unreachable hardware (one AMDXDNA HW context). `capabilities.toml` save/load refuses this state hard. PR-20 ships the validator + a clear error in the dashboard.
- **`route_to_chat` between two NPU `llm` slots is unreachable.** Mid-conversation persona swap that crosses NPU chat models means an FLM process swap mid-stream. Blocked at OmniRouter dispatch; no exception path.
- **Future FLM roles (e.g., reranking).** The trio architecture extends to a quartet naturally — add a fourth coresident slot, add `--rerank 1` (or whatever flag FLM ships) to `flm.args`. No ADR amendment needed for adding roles in the same shape; ADR amendment IS needed if FLM ever lets us run two AMDXDNA HW contexts (which would invalidate Layer 1's "one process" premise entirely).
