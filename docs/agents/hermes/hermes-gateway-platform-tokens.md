# Hermes gateway ŌĆö platform tokens (SYSTEM scope)

How the **Hermes gateway** (`hermes-gateway.service`) gets its Telegram +
Discord bot tokens and other platform credentials, and why they live in a
systemd **drop-in** rather than the main unit.

This is the runbook for issue [#437](https://github.com/Hal0ai/hal0/issues/437),
rescoped from user-scope to **system-scope**: the gateway runs as
`/etc/systemd/system/hermes-gateway.service` with `User=hal0`,
`WantedBy=multi-user.target` (the same `hal0` identity the dashboard unit
`hal0-agent@hermes.service` uses). There is no user bus and no
`XDG_RUNTIME_DIR` to worry about ŌĆö everything below uses plain
`systemctl`, never `systemctl --user`.

## The two surfaces

| Unit | Scope | What it is |
|---|---|---|
| `hal0-agent@hermes.service` | system (`hal0-agent@.service` template) | the **dashboard** (`hermes dashboard --tui`), `/api/events` + `/api/pty` |
| `hermes-gateway.service` | system (`/etc/systemd/system/hermes-gateway.service`) | the **gateway** ŌĆö the Telegram/Discord bot |

Both run as `User=hal0`. This page is about the **gateway**.

## Where the tokens live

### 1. The secrets vault (root:root 0600)

```
/var/lib/hal0/secrets/agents/hermes.env
```

Owner `root:root`, mode `0600`. Under a **system** unit, PID 1 (root)
reads the `EnvironmentFile` before dropping to `User=hal0`, so the file
can ŌĆö and should ŌĆö stay root-only. The `hal0` service account never
needs to read it directly.

Keys it holds:

| Key | Purpose |
|---|---|
| `OPENROUTER_API_KEY` | model routing for the gateway |
| `TELEGRAM_BOT_TOKEN` | Telegram bot auth |
| `TELEGRAM_ALLOWED_USERS` | comma-separated allow-list |
| `TELEGRAM_HOME_CHANNEL` | default Telegram channel |
| `DISCORD_BOT_TOKEN` | Discord bot auth |
| `DISCORD_ALLOWED_USERS` | comma-separated allow-list |
| `DISCORD_HOME_CHANNEL` | default Discord channel |
| `FAL_KEY` | fal.ai media generation |
| `GATEWAY_ALLOW_ALL_USERS` | open-access override (use with care) |

The hal0 provisioner writes/merges this vault in its `voice_wire` phase
via `_merge_env_file` (preserving operator-added lines, atomic write,
chmod 0600).

### 2. The systemd drop-in (root:root 0644)

```
/etc/systemd/system/hermes-gateway.service.d/10-hal0-secrets.conf
```

Owner `root:root`, mode **`0644`** (world-readable, like any normal unit
fragment ŌĆö the *secrets* are in the 0600 vault it points at, not here).
Contents:

```ini
# hal0-managed (issue #437) ŌĆö DO NOT EDIT BY HAND.
# ...
[Service]
EnvironmentFile=/var/lib/hal0/secrets/agents/hermes.env
```

## Why a drop-in, not a main-unit edit

The main unit `hermes-gateway.service` is generated by
`hermes gateway install --system --run-as-user hal0`
(`hermes_cli/gateway.py:generate_systemd_unit`). hermes_cli's
`refresh_systemd_unit_if_needed` **rewrites the `.service` body** whenever
the installed unit drifts from what the generator emits ŌĆö so any
`EnvironmentFile=` baked into the main unit by hand gets **clobbered** on
the next regeneration, and Telegram/Discord silently go offline.

A drop-in under `ŌĆ”/hermes-gateway.service.d/` is never touched by that
regeneration: systemd merges `.d/*.conf` on top of the main unit, and
hermes_cli only owns the `.service` file, not the `.d/` tree. So the
token wiring survives every `hermes gateway install`.

This is the core of #437: **fresh provisions/reinstalls come up with
Telegram + Discord connected, and stay connected across main-unit
regeneration.**

## How it gets installed (idempotently)

The hal0 provisioner owns the drop-in in its **`gateway_secrets_wire`**
phase (`src/hal0/agents/hermes_provision.py`,
`_phase_gateway_secrets_wire`), which runs after `voice_wire` and before
`smoke_tests`:

1. `mkdir -p /etc/systemd/system/hermes-gateway.service.d` (mode 0755).
2. Render the drop-in body (deterministic ŌĆö header + `EnvironmentFile=`).
3. Compute a content hash; **if the on-disk file already matches, skip
   the write AND skip `systemctl daemon-reload`** (idempotency, mirrors
   `config_write` / `_merge_env_file`).
4. Otherwise write atomically (tmpfile + `os.replace`), `chmod 0644`.
5. Run `systemctl daemon-reload` **only when the file changed**.
6. Non-root provisions `SKIP` with a clear reason (can't write
   `/etc/systemd/system`).

The phase deliberately does **not** generate the main unit ŌĆö that's the
orchestrator's `hermes gateway install` step during cutover. Keeping unit
generation out of the phase avoids the generator's custom-`HERMES_HOME`
trap and keeps the blast radius small.

## Manual re-apply

If you edit the vault or need to force a re-wire by hand:

```bash
# (re-)write the vault ŌĆö operator edits to non-hal0 keys are preserved
sudo vim /var/lib/hal0/secrets/agents/hermes.env

# re-apply the drop-in + reload + restart
sudo systemctl daemon-reload
sudo systemctl restart hermes-gateway
```

The drop-in itself is idempotent: re-running `hal0 agent bootstrap hermes`
re-writes it only if the content drifted.

## Verification

```bash
# 1. The unit picks up the EnvironmentFile:
systemctl show hermes-gateway -p EnvironmentFiles
#   EnvironmentFiles=/var/lib/hal0/secrets/agents/hermes.env (ignore_errors=no)

# 2. The token actually reached the running process env:
tr '\0' '\n' < /proc/"$(systemctl show -p MainPID --value hermes-gateway)"/environ \
  | grep TELEGRAM_BOT_TOKEN

# 3. The gateway reports its adapters connected:
cat /var/lib/hal0/.hermes/gateway_state.json   # telegram + discord "connected"
```

(`gateway_state.json` lives under the canonical `HERMES_HOME` ŌĆö
`/var/lib/hal0/.hermes`.)

## Restart durability

Because the credentials come from a drop-in `EnvironmentFile`, they
survive:

* `systemctl restart hermes-gateway` ŌĆö re-reads the drop-in every start.
* `hermes gateway install` regenerating the main unit ŌĆö the `.d/` tree is
  untouched.
* a fresh `hal0 agent bootstrap hermes` ŌĆö the phase hash-skips when the
  drop-in is already correct.

## Notes

* Use plain `systemctl` ŌĆö this is a **system** unit. There is no
  `systemctl --user` and no `XDG_RUNTIME_DIR`/user-bus step (that was a
  user-scope artifact of the original #437 framing; dropped here).
* If Discord crashes the gateway on start with
  `discord.errors.Forbidden 403 (50001) Missing Access`, the bot was
  removed from a configured channel (usually `DISCORD_HOME_CHANNEL`).
  That is a token/permissions problem, not a wiring problem ŌĆö fix the
  channel or the bot's membership, it is independent of this drop-in.
