A fresh hal0 install boots into the **FirstRun wizard**. Open the
dashboard at `http://localhost:8080` and the wizard owns the screen
until you've either installed a bundle or chosen to skip and configure
slots by hand.

## What the wizard does

The wizard is a guarded route. Until the API reports `first_run: false`
every other dashboard navigation redirects back to it. There's no point
operating an empty box.

It is a **bundle picker**, not a single-model sequence. Instead of
walking you model-by-model through chat → embed → voice, it offers a
handful of curated **bundles** (read live from `/api/install/bundles`
via `useCuratedBundles()`). Each bundle is a hardware-tiered loadout —
pick one and hal0 runs an **orchestrated multi-slot install**: it
registers the slots that tier defines and pulls every model they need
in one pass.

## The steps

<Steps>
1. **Pick a bundle.** The picker detects your **RAM, GPU, and NPU** and
   highlights the highest tier that fits as ★ recommended; tiers that
   need more memory than you have are marked `needs ≥ N GB` and
   disabled. Each card (or column, in the table layout) shows the RAM
   floor, the download size, and how many of the 8 slots it brings up.
   You can switch tiers freely — the recommended one is just
   pre-selected.

2. **Set model storage.** A storage field below the picker decides
   **where models live** before any pull starts. It is pre-filled with
   the effective store (e.g. `/var/lib/hal0/models`) and offers
   suggestion chips for detected candidates. Every slot container
   points at this one path, so you pick it once. (If you point it at a
   directory that already holds models, the wizard offers to migrate
   them.)

3. **(Optional) Advanced — NPU & per-slot.** An **Advanced** drawer
   ([PR #812](https://github.com/Hal0ai/hal0/pull/812)) lets you tune
   the install before it runs:
   - **Enable NPU trio** — bring up the agent + STT/embed passengers on
     the NPU (requires an NPU).
   - **Per-slot overrides** — a small table with one row per slot
     (Chat, Coder, Embed, Speech-to-text, Text-to-speech). For each you
     can type a **curated model id** and/or pick a **profile** from the
     dropdown. Blank rows keep the bundle's smart default. The chosen
     profile carries a coherent device (a ROCm profile installs a ROCm
     device, etc.), and overrides only apply to slots the chosen tier
     actually installs.

4. **Install.** Hitting **Install &lt;tier&gt;** POSTs to
   `/api/install/apply` with the tier, storage dir, NPU opt-in, and any
   overrides. The wizard advances to a progress view.

5. **Watch the pulls.** Each model the install kicked off gets a
   download row that reattaches to its live SSE stream — bytes,
   percent, speed, ETA, with per-row **Retry** on failure. Downloads
   continue in the background, so you can hit **Continue** without
   waiting on multi-GB pulls to finish.

6. **Verify services.** The final step confirms the **agent** and
   **chat UI** are running (with a one-click **Retry** if a unit is
   down), and offers a quickstart card for **ComfyUI** image
   generation. **Open dashboard** marks FirstRun complete and hands you
   the rest of the UI.
</Steps>

## Pre-built kits

Below the tier picker is a **Pre-built kits** section: vendor-blessed,
AMD-curated loadouts that come up with multiple slots ready out of the
box. The flagship is **LMX-Omni-52B-Halo** — a Strix Halo kit (≥ 100 GB
unified RAM) that wires four slots in one shot (a 35B chat model,
Whisper-Large STT, kokoro TTS, and Flux-2-Klein image gen), roughly a
75 GB download. Picking a kit runs the same orchestrated multi-slot
install as a tier.

<Aside type="note" title="Skipping the wizard">
You don't have to use the wizard. Hit **Skip — configure manually** to
go straight to an empty dashboard and wire slots yourself, or drive it
from the CLI:

```sh
hal0 slot load chat --model phi-3-mini-4k-instruct-q4
```

The slot manager handles the pull, the systemd unit, and the health
probe. `hal0 status` shows the same state machine the wizard streams.
</Aside>

## When a pull fails

Each download row surfaces its own error inline and offers **Retry** to
re-fetch. Common causes:

- **No disk space.** The storage path you chose ran out mid-pull. Free
  the space and retry — partial downloads resume.
- **Hugging Face rate-limit.** Anonymous pulls hit a rate cap on
  popular weights. Set `HF_TOKEN` (in the dashboard under Settings →
  Secrets, or in `/etc/hal0/api.env`) and retry.
- **License not accepted on Hugging Face.** Some gated models require
  acceptance on the HF side before the API will serve the files. Accept
  on the model page, then retry the row.

A failed pull leaves its slot in `error` with details until you retry;
nothing is hidden, and the other slots in the bundle continue
independently.

## Picking something other than the bundle

The bundle is a starting point. After FirstRun the **Models** page in
the dashboard is where you live:

<img
  src="/screenshots/models-registry.png"
  srcset="/screenshots/models-registry.png 1x, /screenshots/models-registry@2x.png 2x"
  alt="Models registry table at /models showing rows with chat / STT / TTS / rerank capability tags, on-disk size, and the backend that owns each model."
  width="1440"
  height="900"
/>

Each row carries the model's capability tags, on-disk size, and the
backend that owns it. From here you can pull more weights, assign a
model to a slot, or hand the same job to the CLI:

```sh frame="terminal"
hal0 model list
hal0 slot swap chat --model qwen2.5-coder-7b-instruct-q4_k_m
```

See [recommended loadouts](/docs/hardware/strix-halo/#recommended-loadouts)
for a hardware-by-hardware breakdown of what fits where.
