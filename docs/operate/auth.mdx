hal0 binds the API on `:8080` and OpenWebUI on `:3001` with **no
authentication and no TLS**. This is the right posture for the common
case: an LXC sitting behind your homelab gateway, reached only over the
LAN or through your existing Traefik / nginx / Cloudflare Tunnel vhost
that already does the auth + TLS work.

There is no built-in option to add auth or TLS from inside hal0
itself. The installer no longer bundles Caddy and no longer mints a
first-run wizard password. See [ADR-0012](https://github.com/Hal0ai/hal0/blob/main/docs/internal/adr/0012-remove-auth-and-caddy.md)
for the rationale.

<Aside type="caution" title="LAN vs. WAN">
On a trusted LAN with no port forwards, the default no-auth posture
is fine. If hal0 is reachable from anything you don't physically
control (the public internet, a shared VPN, a guest VLAN), put a
reverse proxy in front. There is no middle ground.
</Aside>

## Recommended patterns

<Tabs>
<TabItem label="Traefik">

If you already run Traefik (typical homelab pattern), add a router +
service for hal0 with whatever auth middleware you prefer. Example
dynamic config:

```yaml title="/opt/gateway/traefik/dynamic/hal0.yaml"
http:
  routers:
    hal0:
      rule: "Host(`hal0.example.com`)"
      service: hal0
      entrypoints: [websecure]
      tls:
        certResolver: letsencrypt
      middlewares: [hal0-auth]

  services:
    hal0:
      loadBalancer:
        servers:
          - url: "http://10.0.1.142:8080"

  middlewares:
    hal0-auth:
      basicAuth:
        users:
          - "you:$apr1$..."   # generate with `htpasswd -n`
```

Traefik handles TLS termination, basic-auth, and the cert renewal
lifecycle. hal0 sees plain HTTP on its `:8080` and treats every request
as authenticated by virtue of the upstream proxy.

</TabItem>
<TabItem label="nginx">

```nginx
server {
    listen 443 ssl;
    server_name hal0.example.com;

    ssl_certificate     /etc/letsencrypt/live/hal0.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/hal0.example.com/privkey.pem;

    auth_basic           "hal0";
    auth_basic_user_file /etc/nginx/htpasswd-hal0;

    location / {
        proxy_pass         http://10.0.1.142:8080;
        proxy_http_version 1.1;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        # SSE / WebSocket
        proxy_buffering    off;
        proxy_read_timeout 1h;
    }
}
```

</TabItem>
<TabItem label="Cloudflare Tunnel">

If you want zero-port-forward access, run `cloudflared` on the same
host as hal0 and pair it with a Cloudflare Access policy.

```yaml title="~/.cloudflared/config.yml"
tunnel: <YOUR-TUNNEL-UUID>
credentials-file: /etc/cloudflared/<UUID>.json

ingress:
  - hostname: hal0.example.com
    service: http://localhost:8080
  - service: http_status:404
```

Then add an Access policy in the Cloudflare dashboard scoped to
`hal0.example.com` (e.g. email allowlist, Google OAuth, GitHub OAuth).
Cloudflare handles TLS + auth at its edge; hal0 itself stays open on
loopback.

</TabItem>
</Tabs>

## OpenWebUI's own auth

`hal0-openwebui.service` runs OpenWebUI in `WEBUI_AUTH=False` mode by
default — same trusted-LAN posture as the hal0 API.

If your upstream proxy injects a trusted email header
(`X-Forwarded-Email` or similar), you can let OpenWebUI provision a
user from that header by passing `WEBUI_AUTH=True` +
`WEBUI_AUTH_TRUSTED_EMAIL_HEADER=<header-name>` into the openwebui.env
file at install time. Edit `/etc/hal0/openwebui.env`, restart
`hal0-openwebui`, and OpenWebUI will skip its login screen in favor of
the upstream identity.

```bash
sudo $EDITOR /etc/hal0/openwebui.env
# add: WEBUI_AUTH=True
# add: WEBUI_AUTH_TRUSTED_EMAIL_HEADER=X-Forwarded-Email
sudo systemctl restart hal0-openwebui
```
